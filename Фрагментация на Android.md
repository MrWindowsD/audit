### byedpi (hufrea) — Подход «Потоковый прокси»

Этот движок работает на уровне **TCP-стрима**. Он не видит сырых IP-пакетов, он работает с данными, которые ты отправляешь в сокет.

**Как он «режет» (Логика):**

- **Метод:** `split`. Он просто берет твой буфер данных (например, HTTP GET запрос) и вместо одного вызова `send()` делает несколько.
- **Где смотреть в коде:** Файл `desync.c`. Там вся магия.
- **Фишка:** Он может дробить запрос так, чтобы `SNI` (имя сайта) разрывалось между двумя TCP-сегментами. DPI-система обычно ожидает увидеть `ClientHello` целиком в одном пакете. Если он разорван — «до свидания», фильтр пропускает трафик.
- **Для Android:** Поскольку это прокси, тебе **обязательно** нужен «переводчик» (как `hev-socks5-tunnel`), который заберет пакеты из VPN-интерфейса и превратит их в поток данных для `byedpi`.

### zapret (bol-van) — Подход «Пакетный хирург»

Это совершенно другой уровень. Он работает с **сырыми пакетами (Raw Packets)**.

**Как он «режет» (Логика):**

- **Метод:** `disorder`, `fake`, `ttl`. Он не просто рвет данные, он меняет структуру самого IP/TCP пакета.
- **Магия TTL:** Он может отправить «фейковую» часть запроса с маленьким `TTL` (Time To Live). Этот пакет дойдет до DPI (он его увидит и «отравится» ложными данными), но не дойдет до реального сервера. А настоящий пакет пойдет следом.
- **Проблема «Без рута»:** Чтобы так изгаляться над пакетами, Linux-версии нужен `NFQUEUE`. На Android у нас есть только файловый дескриптор от `VpnService`, который выплевывает сырые байты пакета.
- **Решение:** Чтобы запустить логику `zapret` без рута, нам придется встроить внутрь приложения целый **TCP/IP стек в юзерспейсе** (например, `lwIP` или `gvisor/netstack`), чтобы он пересобирал и заново дробил пакеты вручную. Это в 10 раз сложнее реализации `byedpi`.

### SpoofDPI (xvzc) — «Легкий прокси на стероидах»

Проект написан на **Go** и представляет собой классический локальный прокси-сервер.

- **Как он «режет»:** В отличие от `byedpi`, который просто дробит поток, `SpoofDPI` делает упор на модификацию TLS-рукопожатия (`ClientHello`). Он умеет разделять первый пакет данных на части, добавлять случайный паддинг (байты-пустышки) и манипулировать TCP-флагами, отправляя фальшивые ACK.
- **Архитектура:** Это **L4/L5 прокси**. Он не видит IP-пакетов, он видит TCP-соединения.
- **Плюсы для Android:**
    - **Компиляция:** Go отлично собирается под архитектуры Android (arm64, x86). Можно скомпилировать его как общую библиотеку (`.so`) через `gomobile` или как отдельный бинарник.
    - **Современность:** Код на Go гораздо легче читать и модифицировать, чем C-шные портянки `byedpi`.

**Минусы:**

- **Оверхед:** Рантайм Go добавит к твоему APK минимум 10–15 МБ веса и будет потреблять больше оперативной памяти по сравнению с C-решениями.
- **Интеграция:** Чтобы сделать из него «одну кнопку», всё равно придется писать на Kotlin обертку с `VpnService` и использовать «прослойку» типа `tun2socks`.

### DPITunnel-android (nomoresat) — «Нативная классика»

Это, пожалуй, самый архитектурно близкий к твоей цели проект, так как он изначально создавался под Android.

- **Как он «режет»:** Использует комбинацию Kotlin-кода и нативных C++ библиотек для реализации **Desync-атак** и фрагментации на уровне TCP.
- **Архитектура:** Это полноценное **VPN-приложение**. Оно уже содержит в себе реализацию `VpnService` и умеет работать с TUN-интерфейсом.
- **Плюсы:**
    - **Всё включено:** В репозитории уже есть готовая логика создания туннеля без Root-прав. Тебе не нужно искать сторонние «переводчики» типа `hev-socks5-tunnel`, так как работа с пакетами встроена в движок.
    - **Производительность:** Использование C++ для сетевого ядра обеспечивает минимальные задержки и низкое потребление батареи.
- **Минусы:**
    - **Слабая гибкость:** Набор методов обхода (стратегий нарезки) здесь скромнее, чем в `byedpi` или тем более в `zapret`. Если провайдер внедрит новую систему детекции, обновить логику в этом проекте будет сложнее.

### Эволюция подхода: От классических утилит к gVisor Netstack

Анализ популярных решений вроде ByeByeDPI, zapret, DPITunnel и SpoofDPI выявил общую закономерность. Большинство из них полагаются на каскад из нескольких независимых компонентов. Обычно это стандартный интерфейс перехвата трафика, который передает данные стороннему приложению для конвертации сырых пакетов в потоки, а затем уже направляет их в ядро обработки. Такая архитектура отлично работает для утилит командной строки. При переносе на мобильную платформу она создает лишние задержки и усложняет контроль над соединениями.

Для создания локального и масштабируемого мобильного приложения требуется монолитное сетевое ядро. В качестве такого решения выступает архитектура на базе языка Go и интеграции gVisor Netstack, заимствованная из проекта Sing-box.

**Суть технологии** Вместо использования внешних библиотек для связи компонентов, gVisor реализует полноценный сетевой стек внутри пользовательского пространства приложения. Ядро напрямую забирает дескриптор виртуального интерфейса Android и самостоятельно обрабатывает низкоуровневые пакеты.

Процесс работы выглядит следующим образом. Трафик перехватывается системой и попадает в ядро. Встроенный маршрутизатор анализирует пункты назначения с помощью локальных баз геоданных. Обычные запросы отправляются в сеть без изменений. Соединения к целевым ресурсам перенаправляются в специальный локальный узел. Именно там к ним применяются алгоритмы фрагментации TLS-рукопожатий или сегментации TCP-пакетов.

**Ключевые преимущества для мобильного клиента:**

- **Изоляция процессов:** Вся работа с сетью происходит исключительно локально на устройстве без обращения к удаленным серверам.
- **Гибкая конфигурация:** Управление правилами обхода и маршрутизации осуществляется через структурированные текстовые файлы конфигурации. Это позволяет легко реализовать систему автоматического подбора параметров при обнаружении сбросов соединений со стороны провайдера.
- **Масштабируемость:** Базовая структура готова к тонкой настройке маршрутизации без необходимости переписывать логику взаимодействия с операционной системой Android.

**1. Бесшовная и надежная интеграция через GoMobile** Использование `Runtime.exec()` для управления консольными утилитами (как в ByeByeDPI) на современных версиях Android с жесткими ограничениями фоновых задач приводит к нестабильности, проблемам с разрешениями и утечкам файловых дескрипторов. Ручное написание C++ оберток JNI для lwIP требует уникальных компетенций и замедляет разработку. Инструментарий GoMobile/GoBind решает эту проблему на фундаментальном уровне. Он автоматически компилирует Go-код в `.aar` библиотеку и генерирует Java/Kotlin интерфейсы. Для Android-разработчика взаимодействие с ядром выглядит как вызов стандартных методов объекта, что гарантирует стабильный жизненный цикл сервиса и легкий контроль состояния.

**2. Непревзойденная масштабируемость и экосистема** Пользовательский запрос делает явный акцент на **масштабируемости других функций**. Приложение для обхода блокировок никогда не ограничивается только фрагментацией. Полноценный продукт требует реализации защиты от перехвата DNS (поддержка DNS-over-HTTPS, DNS-over-TLS) , умной маршрутизации (разделение трафика: локальные ресурсы напрямую, заблокированные — через туннель или фрагментацию) , поддержки резервных прокси-протоколов (VLESS, Trojan, WireGuard) для случаев, когда фрагментация бессильна. Написать все это с нуля на C/C++ нецелесообразно. Экосистема Go обладает богатейшим набором готовых, протестированных библиотек для работы с сетью. Архитектура Sing-box уже содержит модульную систему, позволяющую подключать правила гео-роутинга и обработчики протоколов как компоненты конвейера. Добавление новой функции в Go занимает дни, тогда как в C++ — недели.

**3. Мощный контроль над трафиком без Root-прав** gVisor Netstack предоставляет уникальный баланс: он работает в пространстве пользователя, принимая данные от VpnService (не требуя Root), но при этом оперирует сырыми IP-пакетами. Это означает, что разработчик имеет доступ к полному контролю над формированием TCP-заголовков. Netstack позволяет реализовать не только прикладную фрагментацию TLS (разделение ClientHello на несколько записей `tls_record_fragment` ), но и самые сложные методы десинхронизации, такие как инъекция фейковых TCP-сегментов с заданным TTL или искаженной контрольной суммой (аналогично `zapret`). Вы можете создать перехватчик (Endpoint) внутри Netstack, который перед отправкой легитимного SYN-пакета генерирует "мусорный" пакет и отправляет его в сеть для обмана DPI.

**4. Энергоэффективность и производительность на мобильных устройствах** Исторические опасения по поводу потребления батареи программами на Go из-за наличия сборщика мусора (GC) более не актуальны для современных реализаций сетевых стеков. Оптимизации пуллинга буферов (Buffer Pooling) в Netstack устранили проблему избыточных аллокаций памяти, радикально снизив нагрузку на CPU. Отсутствие двойной трансляции трафика (которая присутствует в цепочке TUN -> SOCKS -> ByeDPI) сокращает количество контекстных переключений процессора. Правильно настроенные тайм-ауты TCP-соединений в Netstack предотвращают "залипание" радиомодуля смартфона в активном режиме, сохраняя заряд батареи при переходе устройства в Doze Mode.

**5. Перспективы автоматизации (Geneva Engine)** Системы DPI постоянно обновляются, провайдеры внедряют новые эвристики и блокировки. В будущем может потребоваться интеграция систем на базе генетических алгоритмов, таких как _Geneva (Genetic Evasion)_, которые автоматически обучаются находить бреши в конечных автоматах DPI и генерировать новые стратегии фрагментации и инъекции пакетов. Разработка модулей, подобных проекту _Dynam_ , требует гибкой архитектуры с поддержкой графа состояний. Модульный дизайн на языке Go идеально подходит для развертывания динамически загружаемых конфигураций (JSON-политик обхода), позволяя обновлять стратегии обхода "по воздуху", без необходимости выпускать обновления самого Android-приложения.

## **Рекомендации по реализации**

Для создания отказоустойчивого, высокопроизводительного и масштабируемого приложения на базе выбранной архитектуры, рекомендуется придерживаться следующего инженерного плана и лучших практик.

### **Фаза 1. Проектирование Android-моста и интеграция VpnService**

Основа приложения должна быть написана на Kotlin. Ее главная задача — взаимодействие с системой, управление пользовательским интерфейсом и жизненным циклом службы.

1. **Создание службы VpnService:** Реализуйте сервис, наследующий `android.net.VpnService`. Используйте `VpnService.Builder` для настройки виртуального интерфейса. Назначьте внутренний IPv4/IPv6 адрес (например, `10.0.0.1`), укажите DNS-серверы (маршрутизируя все запросы 53 порта в ваше ядро для защиты от DNS-утечек).
    
2. **Сплит-туннелирование (Per-app Proxy):** Воспользуйтесь методами `addAllowedApplication()` и `addDisallowedApplication()`. Это критически важная функция для обхода ограничений: трафик локальных банковских приложений или мессенджеров должен идти напрямую, в обход виртуального интерфейса, чтобы снизить нагрузку на ядро фрагментации и избежать блокировок по факту использования подозрительных IP-адресов.
    
3. **Передача файлового дескриптора:** Метод `builder.establish()` вернет `ParcelFileDescriptor`. Именно его числовой идентификатор (File Descriptor, FD) необходимо передать в Go-библиотеку.
    
4. **Компиляция ядра GoMobile:** Определите четкий API в Go-коде (например, функцию `StartEngine(fd int, configJSON string)`). Соберите библиотеку командой `gomobile bind -target=android -tags=with_gvisor,with_quic`. Избегайте использования нестандартных типов Go в экспортируемых функциях, так как GoBind поддерживает ограниченный набор трансляций типов в Java.
    

### **Фаза 2. Конфигурация Netstack и устранение проблем с MTU**

Правильная настройка параметров TCP/IP внутри gVisor Netstack — залог стабильной работы модуля без потерь пакетов и зависаний.

1. **Интеграция канального уровня:** Внутри Go используйте пакет `tcpip.link.fdbased` для создания канального уровня (Data-Link Layer), который будет читать и писать байты напрямую из переданного Android-дескриптора.
    
2. **Управление MTU (Maximum Transmission Unit):** Это самая частая причина отказов в работе VPN-приложений. При отправке трафика в TUN-интерфейс, операционная система устанавливает стандартный MTU (обычно 1500 байт). Однако, поскольку пакеты инкапсулируются, реальный размер пакета на физическом интерфейсе Wi-Fi/LTE может превысить 1500 байт, что вызовет фрагментацию на уровне IP. Фрагментация IP смертельна для алгоритмов обхода DPI, так как провайдерское оборудование собирает фрагменты на сетевом уровне до анализа L7, нивелируя все хитрости. ◦ _Рекомендация:_ Установите MTU интерфейса TUN в `VpnService.Builder` и внутри Netstack на консервативное значение (например, 1360 или 1400 байт), а в случаях нестабильных сотовых сетей — вплоть до DSL-стандартов (например, 576 байт). Также обеспечьте программную корректировку параметра TCP MSS (Maximum Segment Size) в пакетах SYN, чтобы гарантировать, что ни один пакет не превысит установленный MTU.
    

### **Фаза 3. Реализация модулей фрагментации и десинхронизации**

Масштабируемая архитектура подразумевает, что методы обхода применяются декларативно, в виде набора фильтров (middlewares), применяемых к TCP-сессиям.

1. **TLS Record Fragmentation:** Для борьбы с современными ТСПУ и системами фильтрации SNI необходимо реализовать поддержку сегментации записей TLS. Когда Netstack восстанавливает TCP-поток, модуль должен проанализировать первые пакеты на предмет наличия TLS `ClientHello`. Вместо отправки единого блока данных, модуль программно разбивает его на несколько независимых записей (TLS Records), вставляя дополнительные заголовки. Идеально, если граница разбиения проходит ровно по полю расширения SNI (`sni boundary mode`), что заставляет парсеры DPI сбоить. Для повышения эффективности между отправками записей можно добавлять миллисекундные задержки, переполняя кеш ожиданий DPI.
    
2. **Генерация фиктивных пакетов (Fake Packets) и TTL:** Для обхода наиболее агрессивных DPI, блокирующих соединения по IP-адресам или поведенческим сигнатурам, необходимо реализовать подсистему генерации фейков, вдохновленную проектом Zapret. ◦ Создайте перехватчик в Netstack, который на этапе установки соединения генерирует поддельный TCP-сегмент (содержащий, например, фальшивый SNI). ◦ Главный трюк заключается в программной установке поля TTL в IP-заголовке этого пакета. Определите значение TTL (например, 8-10), при котором пакет гарантированно пройдет через оборудование DPI (находящееся на стороне провайдера), но "истечет" и будет уничтожен магистральными маршрутизаторами до достижения целевого сервера. Сервер получит только реальный запрос, а DPI занесет в лог фиктивную информацию. Архитектура Netstack позволяет манипулировать полями L3-уровня без Root-прав, модифицируя байты перед их отправкой в физический сокет.
    
3. **Обфускация HTTP:** Для нешифрованного трафика следует внедрить манипуляции заголовком Host: смешивание регистра (Mixed Case) и добавление нестандартного количества пробельных символов до и после заголовков.
    

### **Фаза 4. Интеграция DNS и умной маршрутизации**

Простое фрагментирование пакетов бесполезно, если система DPI применяет перехват и подмену DNS-запросов (DNS Hijacking/Poisoning).

1. **Защита DNS:** Встроенный в ядро модуль должен перехватывать все UDP-пакеты, идущие на порт 53. Эти запросы должны обрабатываться внутренним DNS-клиентом, поддерживающим протоколы DNS over HTTPS (DoH) или DNS over TLS (DoT).
    
2. **Подмена IP (Fake-IP):** Для повышения отзывчивости системы целесообразно внедрить механизм Fake-IP. При получении DNS-запроса от Android, модуль мгновенно возвращает виртуальный IP-адрес (например, из подсети `198.18.0.0/15`), а реальное разрешение доменного имени происходит асинхронно или непосредственно перед установкой TCP-сессии.
    
3. **Динамическая маршрутизация:** Ядро должно поддерживать загрузку списков правил (Rule-sets, GeoIP, Geosite). Если домен находится в белом списке (домашний регион), трафик направляется через прямой системный сокет. Если домен заблокирован — трафик направляется в подсистему фрагментации и десинхронизации. Это критически важно для экономии заряда батареи и снижения задержек.
    

### **Фаза 5. Управление конфигурацией и энергоэффективностью**

Масштабируемое приложение должно быть готово к постоянной "гонке вооружений" с цензорами.

1. **Облачная доставка стратегий:** Настройки обхода (размер фрагментов, значения TTL, тип фейковых пакетов, списки узлов DoH) не должны жестко зашиваться в код. Реализуйте JSON-конфигурации, которые приложение периодически загружает с управляющего сервера (аналог механизмов Sing-box и Hiddify). Это позволит менять стратегии обхода для всех пользователей без обновления приложения в Google Play.
    
2. **Оптимизация батареи:** Фоновое поддержание VPN-соединения истощает заряд из-за периодической отправки пакетов проверки активности (Keep-Alive), которые не позволяют радиомодулю смартфона (Cellular Radio) перейти в спящий режим. Внедрите логику обнаружения неактивности: если экран устройства выключен и активных потоков передачи данных нет, модуль должен агрессивно увеличивать интервалы TCP Keep-Alive и UDP-тайм-аутов, позволяя ОС Android переводить устройство в режим Doze. Реализация архитектуры на базе Go и gVisor Netstack, сопровождаемая строгим соблюдением рекомендаций по управлению MTU, глубокой интеграцией механизмов десинхронизации (включая TTL-манипуляции) и надежной защитой DNS-трафика, обеспечит создание передового, масштабируемого и коммерчески успешного продукта, способного эффективно противостоять современным системам DPI на платформе Android.

### Этап 1: Сборка и подготовка ядра (Go)

Первое, что нужно сделать — превратить исходный код Sing-box в Android-библиотеку (`.aar`), которую сможет прочитать Kotlin. Для этого используется утилита `gomobile`.

1. Настраивается окружение: устанавливается Go, Android SDK и Android NDK.
    
2. Пишется небольшая Go-обертка (Wrapper). Сам по себе Sing-box — это CLI-приложение, а нам нужно API. Обертка будет содержать две функции, которые мы "экспортируем" в Kotlin:
    
    - `Start(config string, fd int) error` — принимает сгенерированный JSON и файловый дескриптор TUN-интерфейса.
    - `Stop() error` — корректно завершает работу процесса, освобождая память.
3. Выполняется сборка:Bash
    
    `gomobile bind -target=android -androidapi 21 -javapkg=com.mrwindowsd.fragmentation ./path/to/wrapper`
    
    На выходе получается файл `fragmentation.aar`, который закидывается в папку `libs` Android-проекта.
    

### Этап 2: Перехват трафика в Android (Kotlin)

В Android-проекте создается класс, наследующий `VpnService`. Это стандартный системный инструмент, который не требует Root-прав.

Его задача — поднять виртуальный сетевой интерфейс:

```kotlin
val builder = Builder()
builder.addAddress("172.19.0.1", 30) // Выделяем внутренний IP для локальной сети
builder.addDnsServer("1.1.1.1") // Устанавливаем DNS
// builder.addAllowedApplication("com.google.android.youtube") // Можно фильтровать по приложениям

val vpnInterface = builder.establish() 
val fd = vpnInterface?.fd // Самая важная деталь - файловый дескриптор
```

`fd` (File Descriptor) — это числовой идентификатор, через который операционная система разрешает читать сырые IP-пакеты (Raw Packets) и писать их обратно.

### Этап 3: Связывание моста (gVisor Netstack)

Здесь происходит главная магия. В стандартных реализациях (типа ByeByeDPI) сырые пакеты из `fd` нужно сначала прогнать через сторонние бинарники (SOCKS-туннели), чтобы превратить их в TCP-потоки.

Поскольку мы используем Sing-box, ядро само умеет читать эти сырые пакеты. Мы передаем `fd` в экспортированную функцию `Start()`:

```kotlin
Fragmentation.start(jsonConfig, fd)
```

Внутри ядра gVisor (реализация TCP/IP стека в userspace) "подхватывает" этот файловый дескриптор. Он берет IP-пакеты, сам собирает их в TCP-соединения, извлекает нужные данные (например, TLS ClientHello, по которому DPI определяет, куда ты идешь) и отправляет на этап маршрутизации.

### Этап 4: Оркестрация и конфигурация (Логика приложения)

Ядро запущено и получает трафик. Теперь им нужно управлять с помощью JSON-конфига, который Kotlin генерирует перед запуском.

В конфиге прописывается структура:

1. **Inbound (Вход):** Тип `tun`, куда мы передаем управление нашим `fd`.
2. **Outbounds (Выходы):**
    - `direct-fragmented`: прямой выход в интернет, но с включенными параметрами `tls_fragment` и `tcp_fragment`.
    - `direct-bypass`: прямой выход в интернет без модификаций.
3. **Routing (Маршрутизация):** Правила, использующие GeoSite. Например, если домен относится к `geosite:youtube` или `geosite:twitter` — отправлять пакеты в `direct-fragmented`. Все остальное — в `direct-bypass`.

Когда DPI (система провайдера) пытается проанализировать соединение к заблокированному ресурсу, исходящий трафик из `direct-fragmented` разбивается на мелкие куски. DPI видит лишь бессвязные обрывки TLS-хендшейка, не может собрать их воедино, не понимает, что это за сайт, и пропускает пакеты дальше.

Если текущие параметры размера фрагмента (например, 10-30 байт) перестают работать из-за обновления DPI, Kotlin-приложение просто генерирует новый JSON с другими размерами (например, 1-10 байт) и вызывает `Stop()`, а затем `Start()` с новым конфигом.

Такой подход оставляет всю сложную сетевую инженерию внутри надежного скомпилированного ядра, а на уровне Android ты работаешь только с простым интерфейсом и конфигурационными файлами.

### **⚠️ Критические нюансы, которые нужно держать в голове**

Ты глубоко копнул теорию, но как человек, который будет это реализовывать с нулевым опытом в Android, учти практические грабли:

### **1. Сборка и размер APK**

Сборка Go через `gomobile` потянет за собой весь Go-рантайм. Финальный `.aar` может весить **от 5 до 15 МБ** в зависимости от включённых зависимостей. Для «оптимизатора сети», который люди будут качать на мобильном интернете, это критично. Нужно будет агрессивно вырезать неиспользуемые пакеты и применять UPX-сжатие (хотя оно может замедлить первый запуск).

### **2. Производительность и энергопотребление**

Ты верно отметил, что старый GC в Go был проблемой. Современный сборщик мусора с буферным пуллингом в Netstack значительно улучшил ситуацию, но **нагрузка на батарею всё равно будет выше, чем у легковесных C-решений**, особенно при активном трафике. Тебе обязательно нужно будет внедрить агрессивные тайм-ауты и мониторинг бездействия, как ты и описал в Фазе 5. Идеально — давать пользователю выбор режима «Экономия энергии» / «Максимальная скорость».

### **3. Отладка Go на Android**

Когда что-то пойдёт не так (а оно пойдёт), ты получишь лог в logcat от Go-рантайма. Это читаемо, но потребует навыка. Рекомендую сразу встроить `log`-пакет Go, который пишет в Android-лог, и предусмотреть сбор краш-дампов.

### **4. Ограничения GoMobile**

GoBind поддерживает не все типы Go. Сложные структуры с интерфейсами не экспортируются. Придётся держать API максимально плоским: `Start(configJSON string, fd int)`. Это ограничение, но для твоей задачи оно некритично.

### **5. MTU и MSS Clamping**

Ты сам написал про это — и это абсолютно правильно. Без принудительной установки MTU в 1360–1400 байт и MSS Clamping фрагментированные пакеты будут пересобираться на уровне IP и ломать всю логику обхода. Это один из главных источников «почему у одних работает, а у других нет». Сделай это первым делом после поднятия TUN.

Давай разберем эти пункты с точки зрения того, как мы будем их обходить в процессе разработки.

1. Размер APK и сборка. Использование флагов линковщика `ldflags="-s -w"` при компиляции Go вырежет отладочную информацию и таблицы символов. Это сразу срежет несколько мегабайт. Применять UPX для Android библиотек стоит с большой осторожностью. Современные версии Android иногда отказываются загружать сжатые таким образом библиотеки в память. Также это иногда вызывает ложные срабатывания антивирусов. Лучше полагаться на форматы Android App Bundle. Они доставят пользователю только нужную архитектуру процессора и оптимизируют итоговый вес скачиваемого приложения.
2. Энергопотребление. Это главная плата за использование gVisor. Сборщик мусора Go будет регулярно будить процессор. Идея с переключателем режимов отличная. По умолчанию ядро должно обрабатывать только трафик из списка заблокированных доменов через GeoSite. Все остальные пакеты должны проходить мимо Go-рантайма напрямую в сеть провайдера. Только так можно спасти батарею.
3. Отладка. Искать ошибки сети в общем потоке системного logcat крайне неудобно. Нам потребуется сразу реализовать механизм записи логов ядра Go в отдельный текстовый файл. Он будет храниться во внутренней директории приложения. В интерфейс нужно будет добавить кнопку экспорта этого лога. Это спасет массу времени при разборе проблем.
4. Ограничения GoMobile. Плоский API выступает здесь скорее как преимущество. Вызов `Start(configJSON string, fd int)` заставляет нас делать архитектуру максимально модульной. Вся сложная сетевая инженерия остается внутри ядра. Интерфейс Android занимается только управлением состоянием и генерацией конфигурации.
5. MTU и MSS Clamping. Это критический фундамент. Мобильные операторы часто используют нестандартные размеры пакетов. Если мы установим стандартный MTU 1500 для TUN-интерфейса, пакеты будут дробиться на уровне операционной системы до того, как к ним применится наша логика обхода DPI. Установка MTU на уровне 1350 байт для TUN-интерфейса и настройка MSS Clamping гарантируют доставку наших фрагментированных пакетов до провайдера в неизменном виде.
