## Технические аспекты алгоритмов фрагментации в Sing-box

Проект `sing-box` предоставляет два фундаментально различных метода деконструкции TLS-трафика: `tls_record_fragment` и `tls_fragment`. Эффективное построение конфигурации требует глубокого понимания механики каждого из них, поскольку они оперируют на разных уровнях модели OSI и по-разному воздействуют на анализаторы DPI.

### Фрагментация записей на уровне представления (`tls_record_fragment`)

Метод `tls_record_fragment`, интегрированный в ядро `sing-box` начиная с версии 1.12.0, функционирует на прикладном уровне (уровне представления протокола TLS). Протокол TLS инкапсулирует свои данные в структуры, называемые записями (TLS Records). Каждая запись имеет строго определенный 5-байтовый заголовок, включающий тип контента, версию протокола и длину полезной нагрузки.

При стандартном установлении соединения все данные сообщения `ClientHello` (включая расширение SNI) упаковываются в одну длинную TLS-запись, которая затем оборачивается в один TCP-сегмент. Стратегия `tls_record_fragment` нарушает этот стандартный паттерн. Ядро на языке Go перехватывает буфер `ClientHello` перед его передачей в транспортный уровень и искусственно делит его на несколько частей. Каждая часть снабжается собственным корректным 5-байтовым заголовком TLS.

Таким образом, строка `youtube.com` внутри SNI может быть разорвана: первая TLS-запись будет содержать `you`, а вторая — `tube.com`. Обе записи могут быть отправлены в рамках одного и того же физического TCP-пакета. Системы DPI, использующие оптимизированные алгоритмы поиска (например, регулярные выражения, нацеленные исключительно на первую TLS-запись в потоке), не способны распознать смещенный паттерн и классифицируют соединение как безопасное.

Преимущество данного метода заключается в отсутствии задержек. Поскольку данные не разбиваются на уровне TCP, операционной системе не нужно ожидать подтверждений (ACK), а пакеты передаются с максимальной скоростью. Разработчики архитектуры настоятельно рекомендуют использовать этот метод в качестве первой линии защиты.

### Сегментация на транспортном уровне (`tls_fragment`)

Если система DPI производит глубокую сборку (reassembling) всех TLS-записей перед анализом, метод `tls_record_fragment` становится неэффективным. В таких случаях применяется `tls_fragment`, который оперирует на транспортном уровне (L4), физически разделяя данные на независимые IP/TCP-сегменты. Данный подход концептуально заимствован из методов `split` и `desync`, используемых в утилите `byedpi`.

Механизм `tls_fragment` разбивает буфер с отправляемым `ClientHello` на несколько TCP-сегментов и, что критически важно, внедряет временную задержку между их отправкой. Если отправить сегменты мгновенно друг за другом, существует высокая вероятность того, что стек TCP/IP на стороне клиента, транзитного маршрутизатора или самого оборудования DPI произведет агрегацию пакетов (склеивание) для оптимизации пропускной способности, что восстановит целостность SNI до момента инспекции.

Для предотвращения этого в `sing-box` внедрен параметр `tls_fragment_fallback_delay`. Процесс отправки выглядит следующим образом:

1. Горутина маршрутизатора извлекает первый фрагмент `ClientHello` (зачастую размером от 1 до 3 байт).
2. Вызывается функция `Write` виртуального сокета `gVisor`, которая инкапсулирует байты в TCP-пакет и отправляет его в сеть.
3. Горутина засыпает (`time.Sleep`) на значение, определенное параметром задержки.
4. В это время первый фрагмент достигает оборудования DPI. Машина состояний DPI регистрирует начало сессии, выделяет буфер, но не находит полного имени домена. По истечении доли секунды, столкнувшись с необходимостью освобождать память под сотни тысяч других пользовательских соединений, DPI удаляет контекст этой TCP-сессии.
5. Горутина пробуждается и отправляет оставшиеся фрагменты `ClientHello`. Оборудование DPI, уже забывшее о начале сессии, воспринимает этот пакет как часть уже одобренного или зашифрованного потока данных и не применяет блокировку.

Интеллектуальная составляющая `sing-box` позволяет автоматически вычислять оптимальное время задержки на платформах Linux, macOS и Windows (с правами администратора) на основе RTT (Round Trip Time) до удаленного сервера. Однако в среде Android автоматическое определение часто затруднено ограничениями `VpnService`. В таких случаях используется фиксированное значение `tls_fragment_fallback_delay` (по умолчанию 500 миллисекунд). Если фактическое время ожидания составляет менее 20 миллисекунд, ядро классифицирует целевой узел как локальный прокси-сервер и также переходит к использованию фиксированной задержки, чтобы избежать слишком быстрой передачи фрагментов.

### Архитектура конфигурации маршрутизации

Правильное применение описанных методов требует их строгой изоляции от легитимного трафика. Поскольку `tls_fragment` вносит существенные искусственные задержки (вплоть до 500 мс на этапе рукопожатия), глобальное применение этого метода ко всему трафику устройства приведет к катастрофической деградации пользовательского опыта. В конфигурации `sing-box` в формате JSON для этого используются объекты `route-options` в сочетании с правилами `domain_suffix` или `geosite`.

Ниже приведена структура конфигурации, демонстрирующая правильные блоки маршрутизации:

JSON

`{ "route": { "rules": [ { "domain_suffix": [ "youtube.com", "googlevideo.com", "discord.com", "discordapp.com", "instagram.com", "cdninstagram.com" ], "action": "route-options", "tls_fragment": true, "tls_fragment_fallback_delay": "150ms", "tls_record_fragment": true }, { "domain_suffix": [ "youtube.com", "googlevideo.com", "discord.com", "discordapp.com", "instagram.com", "cdninstagram.com" ], "action": "route", "outbound": "direct-fragmented" } ], "final": "direct", "auto_detect_interface": true }, "outbounds": [ { "type": "direct", "tag": "direct-fragmented" }, { "type": "direct", "tag": "direct" } ] }`

В данной архитектуре правило с `action: route-options` является нефинальным. Оно перехватывает соединения к целевым доменам, помечает внутренний контекст соединения внутри ядра флагами необходимости применения `tls_fragment` и `tls_record_fragment`, а затем передает управление следующему правилу. Второе правило маршрутизирует модифицированный поток в исходящий интерфейс `direct-fragmented`. Начиная с версии `1.12.0-beta.24`, допускается одновременное включение обоих методов фрагментации, что позволяет объединить деконструкцию TLS-фреймов с физическим микро-разрывом TCP-сегментов.

## Интеллектуальный механизм автогенерации и подбора параметров

В связи с тем, что оборудование DPI, алгоритмы эвристического анализа и паттерны фильтрации радикально различаются в зависимости от интернет-провайдера (ISP) и региона , использование статической конфигурации, жестко зашитой в код мобильного приложения, обречено на провал. Стратегия, обеспечивающая 100% обход в сети оператора сотовой связи, может демонстрировать нулевую эффективность в домашней Wi-Fi сети того же пользователя. Для решения этой проблемы требуется внедрение системы динамического профилирования — алгоритма автоматического подбора (Autotuning), который будет инициализироваться при смене сетевого интерфейса или при запуске приложения.

Вдохновением для подобной системы служат аналитические скрипты, такие как `blockcheck.sh` из комплекса `zapret`. Однако, в отличие от bash-скриптов, работающих в интерактивном режиме на ПК, механизм для Android должен быть полностью автоматизирован, асинхронен и интегрирован непосредственно в код на языке Go, взаимодействуя с ядром `sing-box`.

### Матрица профилей обхода (Strategy Matrix)

Алгоритм инициализирует тестирование, перебирая заранее определенную матрицу стратегий. Матрица должна быть упорядочена от наименее интрузивных методов (обеспечивающих максимальную скорость) к наиболее агрессивным (вносящим задержки). Данный подход гарантирует, что алгоритм выберет оптимальный баланс между успешностью обхода и скоростью загрузки ресурсов.

|**ID Профиля**|**Конфигурация стратегии обхода**|**Характер воздействия**|**Ожидаемое влияние на Latency**|
|---|---|---|---|
|**0**|Без вмешательства (Baseline)|Прямое подключение для проверки отсутствия блокировок.|Нулевое|
|**1**|`tls_record_fragment: true`|Фрагментация записей без задержек TCP.|Минимальное|
|**2**|`tls_fragment: true`, Delay: `50ms`|Быстрая TCP-сегментация. Эффективна для локальных DPI.|Низкое|
|**3**|`tls_fragment: true`, Delay: `150ms`|Стандартная TCP-сегментация.|Среднее|
|**4**|`tls_fragment: true`, Delay: `500ms`|Глубокая TCP-сегментация для тяжелых DPI с большим кэшем.|Высокое|
|**5**|Hybrid (`Record` + `Frag 300ms`)|Максимально агрессивная деконструкция пакетов.|Очень высокое|

### Математическая модель оценки эффективности

Приложение оперирует пулом из $N$ критически важных доменов, которые достоверно подвергаются цензуре (например, $N=10$, включая сервисы `youtube.com`, `discord.com`, `instagram.com`). Для каждого профиля из матрицы инициируется серия параллельных запросов ко всем доменам из пула.

Результат тестирования домена $i$ определяется как бинарная функция успешности $S_i \in \{0, 1\}$. Оценка $S_i = 1$ присваивается исключительно в случае успешного завершения полного цикла TLS-рукопожатия (получения сообщения `ServerHello` от удаленного узла) в рамках заданного жесткого тайм-аута. Если соединение сброшено по тайм-ауту, прервано пакетом TCP RST или сервер не вернул корректный сертификат, $S_i = 0$.

Суммарная эффективность профиля $E$ вычисляется как отношение успешных соединений к общему числу доменов:

$$E = \left( \frac{\sum_{i=1}^{N} S_i}{N} \right) \times 100\%$$

Профиль признается валидным, если его эффективность $E$ превышает заданный порог (например, $\ge 90\%$). В случае если несколько профилей достигают 100% эффективности, система автоматически выбирает профиль с наименьшим значением ID, поскольку он обеспечивает наименьшую деградацию пользовательского опыта (Latency).

## Инженерная реализация модуля тестирования на Go

Для обеспечения надежности и скорости оценки (чтобы процесс автоподбора не задерживал запуск VPN на Android более чем на 3-5 секунд), модуль тестирования должен быть реализован на языке Go с использованием конкурентности (Goroutines). Принципиальной ошибкой многих разработчиков является использование стандартного `http.Get` для проверки доступности. Это приводит к загрузке полного тела веб-страницы, расходу трафика и искажению метрик из-за долгой передачи данных. Эффективный сканер (Prober) ограничивается исключительно установлением TCP-соединения и проверкой состояния TLS (так как DPI блокирует трафик именно на этапе `ClientHello`).

### Концепция оркестратора запросов

Разрабатываемый пакет должен генерировать специфические TCP-диалеры (`net.Dialer`), которые оборачивают соединения в логику фрагментации `sing-box`.

Пример архитектурного паттерна на языке Go демонстрирует использование пакета `context` для строгого контроля времени исполнения :

Go

`package autotune

import ( "context" "crypto/tls" "fmt" "net" "sync" "time" // Внутренние библиотеки sing-box для генерации диалеров с поддержкой route-options )

// Config определяет параметры профиля обхода type Config struct { ID int RecordFragment bool TLSFragment bool FallbackDelay time.Duration }

// Result хранит вычисленную эффективность type Result struct { Config Config Efficiency float64 Latency time.Duration }

// checkDomain выполняет попытку завершить TLS-рукопожатие func checkDomain(ctx context.Context, customDialer net.Dialer, domain string) bool { target := fmt.Sprintf("%s:443", domain)

```
// Подключение через кастомный диалер, инкапсулирующий логику фрагментации
conn, err := customDialer.DialContext(ctx, "tcp", target)
if err!= nil {
	// Ошибка TCP (например, DPI сбросил соединение SYN-пакетом)
	return false
}
defer conn.Close()

// Настройка TLS с указанием SNI, провоцирующего DPI
tlsConfig := &tls.Config{
	ServerName:         domain,
	InsecureSkipVerify: true, // Пропуск валидации цепочки сертификатов для ускорения теста
}

tlsConn := tls.Client(conn, tlsConfig)

// Принудительная инициализация Handshake. Именно здесь отправляется ClientHello
err = tlsConn.HandshakeContext(ctx)
if err!= nil {
	// DPI обнаружил SNI и разорвал соединение (RST)
	return false
}

return true
```

}

// EvaluateProfile конкурентно тестирует массив доменов func EvaluateProfile(profile Config, domainsstring) Result { var wg sync.WaitGroup var successes int var mu sync.Mutex

```
// Инициализация внутреннего механизма фрагментации sing-box для данного профиля
dialer := buildSingBoxDialer(profile) 
startTime := time.Now()

for _, domain := range domains {
	wg.Add(1)
	go func(d string) {
		defer wg.Done()
		
		// Выделение 3 секунд на каждую попытку. DPI обычно реагирует за миллисекунды.
		ctx, cancel := context.WithTimeout(context.Background(), 3*time.Second)
		defer cancel()

		if checkDomain(ctx, dialer, d) {
			mu.Lock()
			successes++
			mu.Unlock()
		}
	}(domain)
}

// Синхронизация всех горутин
wg.Wait()
totalTime := time.Since(startTime)

efficiency := (float64(successes) / float64(len(domains))) * 100.0

return Result{
	Config:     profile,
	Efficiency: efficiency,
	Latency:    totalTime,
}
```

Данная архитектура защищена от эффекта "Head-of-Line Blocking", когда недоступность одного сервера замораживает весь цикл тестирования. За счет использования `sync.WaitGroup` и параллельных горутин, время оценки одного профиля равно времени самого медленного запроса (ограниченного тайм-аутом в 3 секунды). После перебора всей матрицы (или достижения профиля с эффективностью 100%), оркестратор сериализует параметры лучшего `Config` в формат JSON, формируя окончательные блоки `route` и `route-options` для интеграции в основное ядро.

## Динамическое применение конфигурации (Hot Reload) на платформе Android

Окончательным вызовом является применение сгенерированного JSON-конфига к активной сессии без прерывания пользовательского соединения. Мобильные платформы обладают высокой чувствительностью к состоянию сетевых интерфейсов. Перезапуск службы `VpnService` для применения новых параметров маршрутизации приведет к временной потере доступа к сети, сбросу всех текущих TCP-соединений ОС и появлению артефактов в пользовательском интерфейсе (мерцание иконки VPN-ключа).

Архитектура современных приложений (таких как `KunBox` или специализированные форки `sing-box`) решает эту проблему посредством взаимодействия с разделяемой библиотекой `libbox`. Модуль `libbox` представляет собой точку входа (API), скомпилированную в формат AAR для Android с использованием инструментария `gomobile`. Это обеспечивает мост JNI (Java Native Interface) между бизнес-логикой Kotlin на Android и внутренними сервисами ядра на Go.

Механизм «Горячей перезагрузки» (Hot Reload) функционирует следующим образом:

1. **Первичная инициализация:** При нажатии кнопки "Подключить" в приложении, интерфейс `VpnService` поднимается мгновенно с использованием консервативного (базового) профиля `sing-box`, не содержащего тяжелых алгоритмов фрагментации. Это обеспечивает мгновенный доступ к неблокируемым ресурсам.
2. **Фоновый анализ:** Одновременно с этим, в изолированном фоновом потоке запускается модуль автоподбора параметров на Go (Prober). Он выполняет матричные тесты, не затрагивая основной маршрутизатор TUN-интерфейса.
3. **Генерация и Инъекция:** После выявления профиля со 100% эффективностью (например, профиль ID 2: `tls_fragment` с задержкой 150 мс), оркестратор генерирует обновленный JSON.
4. **Атомарная подмена:** Сгенерированная строка JSON передается в метод ядра через JNI (например, вызовом функции `Reload` или `UpdateConfig` внутри `libbox`). Ядро `sing-box` спроектировано таким образом, что оно способно перестроить внутренние деревья маршрутизации и обновить параметры `route-options` в оперативной памяти без уничтожения родительского процесса и без закрытия дескриптора `tun_fd`.

Все новые исходящие TCP-запросы, инициируемые системными приложениями к адресам `youtube.com` или `discord.com`, будут мгновенно перехватываться обновленными правилами и фрагментироваться по выверенному алгоритму. Уже открытые и установленные соединения не прерываются, сохраняя свой контекст и обеспечивая полностью бесшовный пользовательский опыт.

Резюмируя, интеграция `gVisor Netstack`, глубокое понимание параметров фрагментации на уровне записей и транспортном уровне, а также разработка интеллектуального механизма эвристического автоподбора превращают мобильное приложение из статического инструмента маршрутизации в адаптивный программный комплекс, способный автономно противостоять динамически меняющимся угрозам со стороны систем глубокого инспектирования трафика.

Приложение определяет, где оно находится.

- **Тип сети:** Mobile (MCC/MNC оператора) или WiFi (SSID).

Вместо последовательного перебора 10 профилей (что долго), мы используем преимущество Go — горутины.

1. Запускаем 3-4 изолированных инстанса сетевого стека внутри приложения с разными стратегиями (от «Light» до «Extreme»).
2. Каждый инстанс пытается выполнить `TLS Handshake` (без загрузки данных) до топ-ресурсов (YouTube, Discord).
3. **Критерий победы:** Тот профиль, который первым и без ошибок установил 3-5 соединений, назначается «Мастером».

### 2. Техническая реализация фрагментации (Deep Dive)

Чтобы «Сделать хорошо» работало на 100%, мы используем два рычага в Sing-box/gVisor.

### А. `tls_fragment` (L4 - TCP Level)

Мы заставляем стек gVisor разбивать поток данных на уровне сегментов.

- **Принцип:** Мы передаем в Sing-box параметры вроде `size: 1-10, sleep: 2-5`.
- **Что происходит:** Первый пакет ClientHello уходит размером всего в пару байт. DPI видит начало TCP-сессии, но не видит SNI (имя сайта). Пока он ждет следующие байты, мы делаем микро-паузу (`sleep`), и досылаем остальное. Многие DPI «сдаются» и перестают анализировать поток, чтобы не создавать задержек в сети.

### Б. `tls_record_fragment` (L7 - TLS Level)

Это наше главное преимущество перед ByDPI. Мы режем не просто TCP-пакеты, а сами TLS-записи.

- **Принцип:** Одна логическая запись TLS (например, Handshake) упаковывается в два или более TLS-фрейма.
- **Эффект:** Для сервера это легитимный TLS, он их склеит. Для DPI это выглядит как странная последовательность данных, где SNI разорван посередине.

### 3. Интеллектуальный генератор конфигураций

Чтобы не ограничиваться 8 профилями, мы внедрим механизм «мутации» параметров JSON.

**Логика весов (Твой 0-100% механизм):**

1. **Список целей:** YouTube (вес 40%), Discord (30%), Instagram (30%).
2. **Генерация:** Берем базовый конфиг и добавляем рандомное смещение (Jitter) к параметрам фрагментации.
    - _Пример:_ `size: "1-10"` превращается в `size: "2-12"`.
3. **Тест:**
    - Если YouTube 100%, но Discord 0% — конфиг получает 40 баллов.
    - Если всё работает — 100 баллов.
4. **Оптимизация:** Приложение запоминает: «Для провайдера X в 18:00 лучше всего сработал конфиг с `tls_record_fragment` на 150-м байте». Это становится новым стандартом для этого пользователя.

### 4. Почему твой подход с gVisor побеждает ByDPI

ByDPI — это отличный инструмент, но он работает как «внешний обрезальщик». Он перехватывает пакеты и пытается их модифицировать.

Твоя архитектура на Go/gVisor — это **полноценная замена сетевого стека**.

|**Функция**|**ByDPI (C/Kotlin)**|**Проект "Виски" (Go/gVisor)**|
|---|---|---|
|**Уровень контроля**|L3/L4 (IP/TCP)|L3 - L7 (Полный стек)|
|**Стабильность**|Может рвать соединения|Нативная поддержка TCP-стейтов|
|**TLS-фрагментация**|Ограничена|Полная (Record splitting)|
|**Автоподбор**|Сложно реализовать гибко|Родной для Go (параллелизм)|
|**Прозрачность**|Виден как VPN|Может мимикрировать под системный трафик|

### Итог для разработки:

Для «массового пользователя» тебе нужно реализовать в ядре **State Machine** (Автомат состояний):

1. **State IDLE:** Ждем нажатия кнопки.
2. **State TUNING:** Быстрый перебор стратегий в фоне (горутины).
3. **State CONNECTED:** Работа на лучшем конфиге + мониторинг качества.
4. **State ADAPTING:** Если качество упало, незаметно для пользователя подкручиваем параметры.

Такой подход сделает «Виски» не просто инструментом, а «умным щитом», который живет своей жизнью и просто дает пользователю доступ к интернету без лишних вопросов.

# **Технический анализ фрагментации и автотюнинга**

Этот документ описывает внутреннее устройство фрагментации в Go-ядре (на базе gVisor) и логику работы системы автоматического подбора параметров для обхода DPI.

## 1. Механизмы фрагментации в Sing-box / gVisor

В современных реализациях (таких как `sing-box` или `amnezia-wg`) фрагментация происходит на уровне пользовательского стека TCP/IP (User-space stack). Использование **gVisor Netstack** позволяет нам игнорировать системные настройки TCP и напрямую управлять тем, как байты упаковываются в сегменты.

### А. TCP Fragmentation (tls_fragment)

Это разделение полезной нагрузки (Payload) на уровне TCP-сегментов.

- **Принцип:** Когда приложение отправляет TLS ClientHello (обычно ~200-500 байт), мы не пишем его одним вызовом `Write()`. Мы разбиваем его на части, например, `[1, 5, 500]`.
- **Технический нюанс:** DPI-системы часто ожидают увидеть SNI в первом или втором пакете. Если мы отправим 1 байт в первом пакете, 5 во втором, и остальное в третьем, многие DPI не смогут собрать «пазл» в реальном времени из-за высокой нагрузки на их собственные буферы.
- **В gVisor:** Мы используем кастомный `Dialer`, который оборачивает соединение и принудительно ограничивает размер первого/второго сегмента (MSS Clamping или явный `Write` по частям).

### Б. TLS Record Fragmentation (tls_record_fragment)

Это более «хитрый» метод, работающий на уровне протокола TLS (L7), а не TCP (L4).

- **Принцип:** Протокол TLS позволяет разбивать одно сообщение (например, ClientHello) на несколько **TLS Records**. Каждый рекорд имеет свой заголовок (5 байт).
- **Почему это работает:** Большинство DPI-движков умеют парсить только первый TLS Record. Если мы разобьем ClientHello так, чтобы часть SNI оказалась во втором рекорде, DPI увидит «неполные данные» и пропустит пакет.
- **Реализация:** Требует ручной сборки TLS-фреймов. Мы берем оригинальный ClientHello, режем его по смещению (например, на 100-м байте) и оборачиваем каждый кусок в стандартный заголовок TLS `0x16 0x03 0x03 ...`.

## 2. Механизм автогенерации конфигураций (Tuning Engine)

Твоя идея с эффективностью в % — отличный базис. Чтобы сделать это гибким, мы внедрим **Генетический Алгоритм** или **Линейный Поиск по приоритетным профилям**.

### Логика работы тюнера:

1. **Пул стратегий (Базовые гены):**
    - `Strategy A`: Только TCP-фрагментация (размеры: 1, 5, 10).
    - `Strategy B`: Только TLS Record фрагментация.
    - `Strategy C`: Смешанный режим + Random Padding.
    - `Strategy D`: Изменение порядка следования полей в TLS (если поддерживается).
2. **Процесс тестирования (The Gauntlet):**
    - При запуске приложение выбирает «Быстрый тест» (3 топовых сайта: Google, YT, Discord).
    - Для каждой стратегии запускается `Headless Dial` (попытка установить TLS-соединение без скачивания контента).
    - **Метрика успеха:** `SuccessRate = (ConnectedSites / TotalSites) * 100`.
    - **Метрика скорости:** Время установки TLS Handshake (TTH).
3. **Интеллектуальный подбор:**
    - Если `Strategy A` дает 100%, но TTH высокий, пробуем уменьшить агрессивность фрагментации для экономии ресурсов.
    - Если `Strategy A` дает 0%, немедленно переходим к `Strategy B`.
    - **Параметризация:** Вместо жестких профилей мы генерируем JSON «на лету», где `fragment_size` — это случайное число в диапазоне, который показал лучший результат в тестах.
4. **Адаптивность к сети:**
    - Результаты кешируются для конкретного типа сети (WiFi SSID или Mobile Operator MCC/MNC). При смене вышки или переходе на WiFi тест запускается заново в фоновом режиме.

1. **UI:** Запрашивает статус соединения.
2. **Tuner Service (Go):** * Берет список доменов.

3. **Sing-box Core:** Принимает новый конфиг и перезапускает сетевой стек с минимальной задержкой.
